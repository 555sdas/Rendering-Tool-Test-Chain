# 未来展望与演进规划

## 一、前端一键启动测试

**现状：** 用户需要在 Unity Editor 中手动点击 Start Collection，然后切换到前端查看结果。

**目标：** 在前端项目管理页面点击"启动测试"，后端自动调度 Unity 完成全流程，无需手动操作。

### 技术路线

```
前端点击"启动测试"
  → 后端创建测试任务 → 更新状态为 running
  → 后端 subprocess 启动 Unity（-batchmode 或封装后的可执行程序）
  → Unity 自动采集数据 → 自动上传至后端
  → 程序退出 → 后端标记任务完成
  → 前端轮询/WebSocket 实时刷新显示结果
```

### 关键实现

| 层面 | 工作内容 |
|------|----------|
| 前端 | 项目详情页增加"启动测试"按钮，选择场景后发起请求，实时展示测试进度 |
| 后端 | 新增 `/test-tasks/trigger` 接口，`subprocess.Popen` 启动 Unity 进程，维护进程状态 |
| Unity | 支持命令行参数 `-projectId` `-sceneId` `-autoStart`，启动即自动采集上传 |

## 二、Unity 程序打包为独立可执行文件

**现状：** 测试需要在安装有 Unity Editor 的设备上运行，门槛较高。

**目标：** 将采集器打包为 Standalone 可执行文件，任何设备直接运行，无需 Unity。

### 打包产物

| 平台 | 产物 | 说明 |
|------|------|------|
| Windows | `XRTester.exe` | 双击或命令行启动 |
| macOS | `XRTester.app` | 独立应用包 |
| Linux | `XRTester` | Headless 无头运行 |

### 改动清单

| 步骤 | 内容 |
|------|------|
| 命令行参数解析 | `XRTestConfig` 在启动时读取 `-projectId`、`-sceneId`、`-autoStart`、`-platformBaseUrl` 等参数 |
| Build Settings | 配置目标平台、分辨率、图形 API |
| 构建产物 | `File → Build Settings → Build` 生成可执行文件 |

所有已有的采集、上传、设备令牌认证逻辑无需改动，打包后完全复用。

### 使用示例

```bash
# 命令行一键启动测试
XRTester.exe -projectId 2 -sceneId 3 -platformBaseUrl http://192.168.1.100:8000/api/v1 -autoStart

# 程序自动完成：启动 → 采集60s数据 → 上传 → 退出
```

## 三、WebSocket 实时通信

**现状：** 前端通过轮询获取测试进度，有延迟且浪费资源。

**目标：** 后端与前端建立 WebSocket 长连接，实时推送测试状态、进度百分比、当前 FPS 等。

### 推送事件

| 事件 | 内容 |
|------|------|
| `test.started` | 测试已启动 |
| `test.progress` | 采集进度 (30/60s)、当前 FPS、CPU/GPU |
| `test.session_created` | 平台会话已创建，返回 sessionId |
| `test.completed` | 测试完成，返回报告链接 |
| `test.error` | 测试异常，返回错误信息 |

前端收到事件后实时更新 Dashboard 状态卡片，无需手动刷新。

## 四、多设备并行测试

**现状：** 同一时间只能测一台设备。

**目标：** 后端管理多个 Unity 进程，前端同时发起多设备测试，结果对比展示。

```
项目 "VR大空间"
  ├── 设备 A (Quest 3) → 采集任务 1 → 结果 A
  ├── 设备 B (Quest 2) → 采集任务 2 → 结果 B
  └── 设备 C (PICO 4)  → 采集任务 3 → 结果 C

前端自动生成三设备 FPS/帧时间/资源占用对比图表
```

---

## 五、持续集成流水线

为每个设备维护一套标准测试：

- **每日构建触发测试** — CI/CD (GitHub Actions / Jenkins) 在每日构建后自动跑基准性能测试
- **回归对比** — 与历史基线对比，自动标记性能退化
- **邮件/消息告警** — 掉帧率超标或帧时间恶化自动通知

## 演进路线图

| 阶段 | 内容 | 优先级 |
|------|------|--------|
| **Phase 1** | 前端一键启动 + Unity 命令行自动运行 | 高 |
| **Phase 2** | Unity 打包为独立可执行程序 | 高 |
| **Phase 3** | WebSocket 实时推送测试进度 | 中 |
| **Phase 4** | 多设备并行 + 结果对比 | 中 |
